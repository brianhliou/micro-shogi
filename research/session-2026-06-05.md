## 0. Outcome — where the project stands

- **Goal:** compute the **complete strong solution (endgame tablebase) of Micro Shogi (4×5
  drop-shogi)** — the next rung above Dōbutsu, the largest strongly-solved drop-shogi.
- **State:** scoping + calibration **complete and committed**. We have a from-scratch,
  tested Rust engine; a validated push-based retrograde solver; the first solved sub-game
  (King+Pawn = draw); a measured per-edge cost (~167 ns); and the full sizing/cost/funding
  picture. Everything cheap is done. What remains is the **big build** (streaming/parallel
  retrograde) and the **funding/co-author** path to pay for the run.
- **Headline sizing (corrected to the index domain):** Micro Shogi is **PB-scale,
  ~$40–70k bare-metal** (~$150–280k cloud) for the working solve — see §3.2. The
  reachable-based ~134 TB / ~$10–15k figures are the *compressed-artifact* and *floor*.
- **Key strategic pivot:** de-risk on **4×4 first** (`../shogi4`, ~19× smaller arrangement
  domain, ~$5–15k) before the 4×5 run.

### 1.2 The index-domain correction (your edit, 87b0091) — the *working* count is bigger

A scalable solver can't index reachable-only (an MPHF over ~5×10¹⁴ keys needs ~4 PB to
*build* — Dōbutsu's 333 MB trick worked only because 2.5×10⁸ keys fit in RAM). The scalable
option is a **combinatorial rank** over each material bucket's placements, which spans **all
arrangements** (legal+illegal+unreachable slots). So the *working* index domain is the
**LR-folded arrangement count ≈ 3.9×10¹⁵**, ~7.8× the folded reachable ~5×10¹⁴.

| | reachable basis (floor / artifact) | **arrangement basis (working solve)** |
|---|---|---|
| W/L/D flat array | 134 TB | **~1 PB** (LR-folded) |
| edge-ops → compute | 1.6×10¹⁶ → ~150 core-yr | **~1.25×10¹⁷ → ~660–1,200 core-yr** |
| cost (bare-metal) | ~$10–15k | **~$40–70k** (cloud ~$150–280k) |

Reachable (~134 TB) stays valid as the **compressed downloadable artifact** (enumerate+pack
post-hoc) and the floor. ns/edge and the SCC/push architecture are unaffected — only the
slot count changes. *(Open: confirm no feasible reachable-only index exists — see §5.)*

### 1.3 Dōbutsu benchmark (the measured anchor)

From `../dobutsu-shogi/solver`, validated vs clausecker: **213,993,386** canonical positions,
pull-based Jacobi, **~75 min, ~7 GB, single-thread**; init −78 (gote win in 78); max DTM 173;
compact tablebase **333 MB**; **only ~1.2% draws**. No-drops ablation: 797,658 positions, draw,
max DTM 37 (drops are what make it deep).

### 1.6 Storage / compute / cost (both bases) — see `cost-model.md`

- **Storage:** reachable W/L/D 134 TB; +DTM 668 TB; +slack ~1 PB. **Arrangement basis ~1 PB
  (W/L/D).** Compressed downloadable artifact **~18–40 TB** (Syzygy 7-piece precedent:
  140 TB → 18.4 TB, ~7.6×; WDL compresses ~10×).
- **Compute:** ns/edge 150 (RAM) → 10,000–100,000 (random disk) — the 100–1000× cliff is the
  central risk. ~100 PB cumulative shuffle (reachable basis). Binding constraint: getting edge
  access to stay sequential/RAM-speed.
- **Cost levers:** W/L/D-first (~8×), stream-and-discard (peak storage ~10–30 TB), bare-metal
  vs cloud (~4–5×), symmetry folding (≤4×), compression (4–8×).

### 1.7 Distribution / hosting

The durable artifact (~18–40 TB compressed) is **not** "thousands/month" — the 7-piece chess
tablebase (18.4 TB) lives free on Internet Archive + torrent + free probe APIs (lichess,
sesse.net). Tiers: (0) headline+PV+certificate (free); (1) practical-subset explorer (GB,
cheap); (2) full compressed table on IA/Zenodo/Academic-Torrents ($0); (3) optional live probe
API (~$108–215/mo). Or compute-verify-discard (publish method + hash, delete bulk).

## 2. Questions answered this session

1. **Dōbutsu state space?** 246,803,167 reachable / 1,567,925,964 upper bound (validated).
2. **Micro Shogi state space?** 3.92×10¹⁵ upper bound (validated) / ~5×10¹⁴ reachable (est).
3. **Explosion to the next rung?** ~2.3M× Dōbutsu; Micro sits between Dōbutsu (solved) and
   Minishogi (unsolved, est-only). Micro is the feasible frontier.
4. **State-space vs game-tree?** Distinct axes; routinely conflated. Shogi's 10²²⁶ is
   game-tree; its state-space is ~10⁶⁸–10⁷¹. Retrograde works on state-space.
5. **Is a complete tablebase feasible / how much?** Yes — **PB-scale, ~$40–70k bare-metal**
   (arrangement basis), weeks–months on ~tens of nodes. Feasible, unlike Minishogi.
6. **What do we do with 134 TB–1 PB?** Compress to ~18–40 TB; host free (IA/torrent) like the
   chess tablebases. Ongoing cost $0–200/mo, not thousands.
7. **What drives compute cost?** `edge-ops × ns/edge × $/core-hr`. edge-ops and $/hr are
   pinned; **ns/edge is the entire 10× spread** (RAM-speed vs disk-random).
8. **What's the architecture risk?** Keeping ~10¹⁶–10¹⁷ value lookups RAM-speed (sequential
   shuffle) vs disk-random (100–1000× cliff). The whole project lives/dies here.
9. **Did we get the ruleset right?** Now **2-source verified** (EN+JA Wikipedia): capture-flip
   promotion, unrestricted either-face drops, checkmate (→king-capture terminal), Ōyama origin.
   Residual: per-piece moves sourced only as "standard shogi equivalents"; repetition default.
10. **How can a tablebase be "partial"?** Drop-shogi has **no clean N-piece tablebase**
    (material is conserved). "Partial" = a smaller sibling game (calibration) or a mate-in-≤D
    cut of the real game (first true subset).
11. **What's the per-edge cost?** **~167 ns** (measured on KP) — confirms the RAM floor.
12. **Can I get research money?** Yes but narrowly; the co-author is the keystone (§1.8).

## 4. Next steps & ideas (the full brainstorm)

### 4.1 The de-risk path (recommended order)
1. **Solve 4×4 (`../shogi4`) completely first.** ~19× smaller arrangement domain, ~$5–15k —
   the smallest *full* drop-shogi solve that exercises the entire production pipeline
   (combinatorial rank index, SCC staging, streaming, checkpointing) at a feasible scale.
   This is the real de-risk; 4×5 follows once 4×4 is in the bag.
2. **Mate-in-≤D partial tablebase of 4×5** — a true subset of the target (run the push
   retrograde D layers deep), useful and correct, the first real slice.
3. **Publish the partial result + validated numbers** → warm-intro artifact for the co-author.
4. **Recruit co-author → ACCESS/HPC + EV** → full 4×5 run → CG 2026 / ICGA paper → distribute.

### 4.2 Further laddering
- **Piece-set ladder** (KP→KPG→KPGS→full): in-RAM only reaches ~KPG; useful for cross-checks
  and growth factors (~240×/piece-pair), not the full solve.
- **Board-size ladder** (3×4 Dōbutsu → 4×4 → 4×5 Micro → 5×5 Mini): the cleanest scaling
  series; 4×4 is the feasible-full-solve rung.
- **Re-run KPG** (~1 hr) for its value if wanted (driver now prints before validation).
- **Cross-check the engine by re-solving Dōbutsu rules** in this framework vs clausecker.

### 4.3 Parallelism (single machine, near-term)
- **Parallelize the move-gen passes** (enumerate/count/fill/audit) with rayon — they're
  embarrassingly parallel over the position array. Would cut the ~1 hr KPG to minutes and the
  in-RAM ceiling up ~1 order. The HashMap is the contention point → use a **sharded/concurrent
  map** (dashmap) or partition keys by hash.
- **Parallel BFS frontier**: process each DTM layer's resolved positions across threads
  (atomic counter decrements on `cnt[]`).
- **Replace the HashMap with a combinatorial rank** → O(1) lock-free indexing, no cache-thrash,
  no contention — the single biggest in-RAM speedup *and* the scalable path.

### 4.4 Concurrency / distributed (the streaming architecture — the H5 build)
- **SCC staging:** condense the material-signature DAG; process bottom-up in topological order;
  **stream-and-discard** solved buckets (peak storage ~10–30 TB not ~1 PB).
- **Partition into key-range shards** sized to node RAM; process a shard in-RAM, emit
  cross-shard edges to outboxes.
- **Push-based label propagation as BSP supersteps**; shuffle keyed by predecessor-shard
  (Kafka topics / object-store partitions / Pregel-style). Each edge once; draws never touched.
- **Unmove generator** for true predecessor-push (vs storing reverse adjacency, which is the
  in-RAM shortcut used this session).
- **Checkpoint/restart:** generation-stamped, idempotent, deterministic replay (multi-week jobs
  *will* hit interruptions — we saw laptop-sleep suspend/resume already).
- **Keep shuffle node-local / same-AZ** (free) — converts the ~100 PB shuffle from a $ line to
  absorbed instance-hours; external sort/merge for sequential (not random) disk I/O.

### 4.5 Optimizations
- ✅ **Push-based counter-BFS** (done) — kills the draw-rescan pathology; the production core.
- **Combinatorial rank index** — replaces HashMap; enables shard-by-bucket; the scalable key
  (inflates domain ~7.8× but is the only thing that scales).
- **W/L/D-first, DTM-second (or never)** — 8× storage, cheaper compute; optimal play recoverable
  by probe-time local search.
- **Stronger symmetry folding** (up to ~4×).
- **Compression** (EGTB RE-pair/Syzygy-style, 4–8×→ ~18–40 TB artifact).
- **Tight value packing** (2 bits W/L/D; ~9–10 bits DTM).
- **mmap value arrays**; **SIMD/bitboard move-gen** (Clausecker's specialty — co-author synergy).
- **Two-bit retrograde then a DTM fill pass** (pipelines I/O).
- **Reduce passes** (combine count+fill; branching computed during solve — done this session).

### 4.6 Correctness / validation
- **Code-impl move-gen diff** (Fairy-Stockfish/hachu) — the pre-publication independent check.
- **Consistency audit** (done) + **push≡pull cross-check** (done) — the rigorous in-house checks.
- **Resolve repetition** rule; brute-force minimax on small buckets; second implementation.

### 4.7 Strategy / publication / funding
- 4×4 de-risk → partial 4×5 → publish → co-author (Clausecker/UAlberta) → ACCESS/EV → full run

## 7. Lessons (transferable, harvested to commit trailers)

- Validate a combinatorial estimate against a known published count before trusting it (the
  hand-wave was ~25× low until the enumerator reproduced Tanaka exactly).
- A "verified" ruleset from one secondary source via a summarizer isn't verified — confirm
  load-bearing/surprising rules against a second *independent* source.
- Above RAM scale, cost is governed by memory-access pattern (sequential vs random), not op
  count — the same work can be 100–1000× apart.
- Retrograde over a draw-heavy game must be **push-based**; pull-based Jacobi re-scans draws
  every round → `O(draws × maxDTM)` (fine at 1% draws, fatal at 70%).
- For a solo/unaffiliated researcher, scale-grade compute funding is affiliation-gated → the
  load-bearing move is recruiting an affiliated co-author (clears the gate *and* the credential).
- Piped/redirected stdout is block-buffered — a hang/kill after a println but before exit
  swallows all prior output. Flush after load-bearing results, before any long/abortable step.
- The index *domain* (what a scalable rank function spans) ≠ the reachable set — sizing must use
  the arrangement count, not reachable, for the working footprint.
